# Спецификация MVP

## Цель

Rental Manager должен быть рабочим пультом контроля аренды: открыть приложение и сразу увидеть, кто должен деньги, где давно не снимались показания, какие расходы надо компенсировать и что нужно сделать сегодня. Не справочник ради справочника, а штука, которая ловит забытые дела раньше, чем они превращаются в пожар.

## Архитектура

- Backend: FastAPI.
- База MVP: SQLite в `data/rental_manager.db`.
- UI: статический HTML/CSS/JS без сборщика.
- Отчёты: Excel через `openpyxl`.
- Тесты: стандартный `unittest`, без отдельного тестового фреймворка.

Такой стек выбран, чтобы проект запускался на Windows одной командой, но его можно было позже перенести на VPS и PostgreSQL.

## Основные сущности

- `RentalObject` — объект аренды: Белый дом, Чёрный дом, Баня.
- `Apartment` — квартира/номер внутри объекта: БД1, ЧД1, Баня 1.
- `Tenant` — жилец и контакты.
- `Lease` — активное или закрытое проживание жильца в квартире.
- `RentCharge` — ежемесячное начисление аренды с двумя частями платежа.
- `UtilityService` — услуга объекта: электричество, вода, газ.
- `Meter` и `MeterReading` — счётчики и показания.
- `Tariff` — тариф с историей действия и ступенями.
- `UtilityBill` и `UtilityBillLine` — коммунальный счёт и строки по квартирам.
- `Expense` — расходы с источником средств и компенсацией.
- `PaymentReceipt` — будущая точка привязки чеков и OCR.

## Аренда

У договора фиксируются:

- дата заезда;
- день платежа;
- сумма на ИП;
- сумма личным переводом;
- залог, место хранения и условия;
- контакты жильца.

Платёж считается закрытым только когда закрыты обе части. Если оплачена только одна часть, статус остаётся `частично`.

Статусы:

- `pending` — ожидается;
- `partial` — оплачено частично;
- `paid` — оплачено;
- `paid_ahead` — оплачено вперёд;
- `overdue` — просрочено;
- `deferred` — согласованная отсрочка.

Отсрочка в интерфейсе задаётся количеством дней. В базе хранится рассчитанная дата окончания отсрочки, чтобы пульт мог напоминать, когда договорённость заканчивается.

### Спорные даты

Если день платежа есть в месяце, используется он. Если дня нет, дата переносится на 1 число следующего месяца.

Пример для 31-го числа:

```text
31.01 -> 01.03 -> 31.03 -> 01.05 -> 31.05
```

Это закреплено тестом, потому что именно такие мелочи потом делают вид, что они не мелочи.

## Коммуналка

Поставщик видит только общий счётчик объекта. Квартирные счётчики нужны для внутреннего распределения между жильцами.

Расчёт:

1. Берём общий расход объекта за период.
2. Применяем ступенчатый тариф объекта.
3. Получаем общую стоимость и среднюю цену единицы.
4. Считаем личный расход квартир.
5. Считаем ОДН: общий расход минус сумма квартирных расходов.
6. Распределяем ОДН только между занятыми квартирами по долям.
7. Пустующие квартиры счёт не получают, их доля перераспределяется между занятыми.

Если сумма квартирных счётчиков больше общего расхода объекта, расчёт останавливается с ошибкой.

Если точных показаний нет, можно разрешить прогноз. Интерполяция и прогноз помечаются предупреждением.

## Расходы

Расход хранит:

- дату;
- объект и квартиру, если применимо;
- категорию;
- сумму;
- источник денег: личные, арендный бюджет, другое;
- способ оплаты;
- описание;
- статус компенсации.

Если расход оплачен личными деньгами, он попадает в список ожидающих компенсации. Компенсация закрывается отдельной кнопкой.

## Отчёты

MVP отдаёт Excel:

- аренда за период;
- коммуналка за период;
- расходы за период;
- долги;
- история по квартире/жильцу.

Отчёты пока простые, но уже формируются из реальной базы, а не из нарисованных таблиц.

## Что пока не сделано

- Telegram-бот;
- OCR чеков;
- автоматическая проверка получателя по реквизитам;
- загрузка файлов чеков и договоров;
- роли пользователей;
- полноценная упаковка в Windows-приложение;
- миграции базы данных.

## Демо-данные

Для проверки всех основных экранов есть скрипт:

```powershell
powershell -ExecutionPolicy Bypass -File .\scripts\seed-demo.ps1 -ResetDemo
```

Он создаёт фиктивных жильцов, разные статусы аренды, коммунальные счета, показания, расходы и подозрительный чек. Записи помечаются `DEMO_DATA`, поэтому демо-набор можно пересоздавать без ручной чистки.
